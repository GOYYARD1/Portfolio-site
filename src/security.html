Threat Model

A proper threat model answers four questions:

  • What are we protecting? (Assets)
  • Who are we protecting it from? (Adversaries)
  • How can they attack? (Attack surface + vectors)
  • How well are we defending? (Controls + assumptions + limits)

Let’s go deeper than the simple bullet list.

Assets (What needs protection)

Even though your site is static HTML/CSS, it still has assets worth protecting:

    A. Core Assets

  • Website integrity (no defacement)
  • professional reputation (no malicious content injected).
  • Confidential data (none intentionally stored, but leakage must be avoided).
  • Host system security (the biggest real-world concern).

    • B. system Assets

  • Local filesystem of the host machine.
  • OS and its services (SSH, web server daemon, logging services).
  • Any future monitoring stack (Wazuh/ELK).

    • C. Operational Assets.

  • Log integrity.
  • Accurate detection data.
  • IR documentation.
  • Uptime/availability during demonstration.
  • Even small projects have meaningful assets.

Threat Actors (Who you are defending against).

We classify them by capability, intent, and opportunity:

    A. Casual Scanners

Automated bots checking for:

  • indexing
  • open ports
  • default credentials
  • common CVEs

Competency: low, Frequency: extremely high, Impact: low → moderate.

    B. Script Kiddies

Use prebuilt tools (sqlmap, dirsearch, nikto, metasploit modules) motivated by:

  • Curiosity.
  • Bragging rights.
  • Defacement.

Competency: low → medium, Frequency: extremely high, Impact: moderate (website integrity + reputation)

    C. Focused Attackers

Unlikely for this project unless:

  • SSH is exposed.
  • I host from home.
  • I use weak passwords.
  • I expose dashboards (ELK, Wazuh, etc).

Competency: medium → high, Impact: high → critical.

    D. Opportunistic Attackers

  • People scanning random IPs for easy wins.
  • Will exploit misconfigurations or weak setups.
  • Won’t do deep targeting unless your site has obvious flaws.

Competency: medium, Impact: moderate → high (if misconfigured server).

Attack Surface (Where they can attack)

Even a static site has an attack surface:

    A. Web Server Layer

  • Misconfigured Nginx/Apache.
  • Indexing enabled.
  • Directory traversal via bad routing.
  • Incorrect permissions.
  • Exposed logs or config files.

    B. Network Layer

  • Exposed open ports beyond HTTP.
  • Weak firewall rules.
  • Directory traversal via bad routing.
  • SSH brute-force attacks.
  • Bad NAT / port forwarding setups.

    C. Host Layer

  • Outdated packages.
  • Weak system hardening.
  • Running unnecessary services.
  • Weak root or user passwords.

    D. Human Layer

  • Mistakes in deployment.
  • Misconfigured firewall.
  • Forgotten test pages.
  • Forgetting to disable dangerous modules.

    E. Content Layer

Even static content can leak:

  • Server version.
  • File metadata.
  • Internal directory names.
  • Comments in HTML.
  • Internal paths.

The attack surface is not the HTML — the attack surface is the hosting environment.
